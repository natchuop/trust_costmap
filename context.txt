2. CURRENT ARCHITECTURE (IMPLEMENTED)
================================================================================

Per enabled robot, lidar_mapper nodes publish:

  /<robot_id>/current_observation_map
  /<robot_id>/local_map
  /<robot_id>/shared_map
  /<robot_id>/combined_map
  /<robot_id>/scan_rviz          (always; Reliable QoS)
  /<robot_id>/map_pose           (from experiment_manager; read-only)

Heavy RViz-only topics (selected robot only):

  /<robot_id>/combined_probability_markers
  /<robot_id>/local_history_markers
  /<robot_id>/shared_history_markers
  /<robot_id>/current_observation_markers
  /<robot_id>/footprint_markers

Map roles:
  - current: latest scan only
  - local: persistent own LiDAR evidence (shared to peers)
  - shared: peer local maps fused in
  - combined: local + shared + current priority

Isolation rules (must keep):
  Mapper must NOT publish to /planning_costmap, /base_map, /map_claims,
  /trust_updates, /planned_path, or /<robot>/cmd_vel.

Pose:
  experiment_manager publishes /<robot_id>/map_pose from its kinematic map pose.
  Mapper prefers map_pose; odom is fallback only.

Launch params of interest:
  enable_lidar_mapping:=true|false
  enable_rviz:=true|false|auto
  rviz_robot_id:=auto|<robot_id>
  gazebo_low_graphics:=true|false     NOTE: name is gazebo_low_graphics, NOT low_graphics
  mapping_publish_rate_hz:=2.0

RViz template:
  config/lidar_mapping.rviz.in
  Placeholders: __ROBOT_ID__, __PEER_SCAN_DISPLAYS__
  Launch expands peers into extra LaserScan displays.

Demo script:
  scripts/run_rviz_demo.sh
  (enable_lidar_mapping + enable_rviz + rviz_robot_id:=auto + gazebo_low_graphics:=false)

   plugin. Map may still load; confidence MarkerArray overlays are a backup.

   that was reversed. If Map breaks again on weak GPU, rely on markers or try

   Full TurtleBot meshes + RViz + multi-robot can cause:
     [GZ-VIS] EXCEPTION ... set_pose ... timed out
     route/goal model create timeouts
   Motion may pause until SYNC SUCCESS. Not a mapper bug.
   Mitigations: gazebo_low_graphics:=true (boxes), or keep shadows off (already).

   After several malicious claims, logs like:
     No route for benign_1: (x,y) -> goal
   Expected experiment behavior, not LiDAR/RViz regression.

4. Early RViz looks empty
   Unknown cells (-1) do not draw. Until LiDAR observes free/occupied cells,
   OccupancyGrid looks blank except goals/path. Wait for robots to move and scan.

5. verify_project.sh Text file busy
   Largely fixed via ast.parse; if it recurs, clear __pycache__ or set
   PYTHONDONTWRITEBYTECODE=1.

6. Reference vs this project
   Reference uses Webots + /map static AMCL map + /robot_N/live_map topics.
   This project uses Gazebo MovingAI world + LiDAR-built maps only (no ground-truth
   /map copied into RViz by design). Blue floor grows as robots explore.

   Some automated topic echoes after timeout 100s failed because the ROS graph

11. LAUNCH INSTANT SHUTDOWN BUG (fixed 2026-07-21 ~22:00)
================================================================================

Symptom (user report):
  - RViz appears to "crash" almost immediately
  - Gazebo also closes; user never sees a usable window
  - Console ends with gazebo SIGINT → SIGTERM → SIGKILL after manager exits
  - With --debug, lots of ProcessExited / SignalProcess noise

What was NOT the primary failure:
  - RViz itself was not segfaulting first
  - Gazebo was not spontaneously dying first

Actual failure chain (confirmed by reproducing user launch):
  1. experiment_manager published RViz GridCells incorrectly:
       AttributeError: 'GridCells' object has no attribute 'points'
  2. Manager process died with exit code 1
  3. launch default shutdown_on_manager_exit:=true fired
  4. Launch sent SIGINT to RViz, lidar_mappers, bridges, then Gazebo
  5. Gazebo was slow to die → escalated to SIGTERM then SIGKILL (exit -9)
  6. User perceived this as "RViz crashed and took Gazebo with it"

ROS 2 Jazzy API note:
  nav_msgs/GridCells fields are:
    header, cell_width, cell_height, cells   ← use .cells
  NOT .points (that is Marker / PointCloud style)

Code fix applied:
  - trust_costmap/experiment_manager_node.py
      publish_rviz_base_grid_cells() uses free.cells / occupied.cells
      publish_base_map() wraps GridCells publish in try/except (non-fatal)
      destroy_node() hardened so shutdown RCLError does not force ugly exits
  - trust_costmap/lidar_mapping_node.py
      publish_lidar_grid_cells() uses .cells.append(...)

Verification after fix:
  - colcon build trust_costmap succeeded
  - ~55s launch with enable_rviz:=true stayed up: no AttributeError,
    no experiment_manager "process has died", RViz OpenGL started,
    lidar_mappers ready, planning Plan #1/#2/#3 continued
